# План активации ML-функций в HTX проекте (WSL2)

## Обзор текущего состояния

После успешной миграции проекта на WSL2, у нас есть возможность активировать ML-компоненты, которые ранее могли быть ограничены или недоступны в WSL1. WSL2 предоставляет лучшую производительность и совместимость с ML-библиотеками.

## Доступные ML-компоненты в проекте

1. **ML API Endpoints** (`/api/v1/ml/`):
   - Планирование ML-экспериментов
   - Пакетная обработка и маркировка данных
   - Поиск по сходству (similarity search)

2. **Сервисы для ML-анализа**:
   - `enhanced_risk_metrics.py` - Расширенные метрики риска
   - `fingpt_service.py` - Финансовая аналитика с использованием FinGPT
   - `mistral_service.py` - Расширенная аналитика с Mistral LLM

3. **Конфигурация ML**:
   - ML_DEVICE: "cpu" (по умолчанию) или "cuda" (если доступна NVIDIA GPU)
   - LOAD_IN_4BIT: True (квантование для оптимизации памяти)
   - TORCH_DTYPE: "float16" (тип данных для вычислений)

## Шаги по активации ML-функций

### 1. Проверка и установка необходимых зависимостей

```bash
# Запустите скрипт для обновления ML-зависимостей
./update_deps_wsl2.sh
```

Основные требуемые пакеты:
- scikit-learn
- pandas
- numpy
- matplotlib
- seaborn
- torch/PyTorch (для LLM моделей)
- transformers (для FinGPT и Mistral)

### 2. Настройка конфигурации ML

Проверьте и, при необходимости, обновите конфигурацию в `backend/app/core/config.py`:

```python
# ML settings
ML_DEVICE: str = "cpu"  # "cuda", если доступна NVIDIA GPU в WSL2
LOAD_IN_4BIT: bool = True  # Для эффективности памяти
TORCH_DTYPE: str = "float16"  # "float32" или "float16"
```

### 3. Включение и тестирование ML API

1. **Тест базовых метрик риска**:
   - Endpoint: `GET /api/v1/ml/risk-metrics?symbol=BTC&days_lookback=30`

2. **Тестирование LLM-интеграции**:
   - Endpoint: `POST /api/v1/ml/analysis/market-sentiment`
   - Body: `{"symbol": "BTC", "period": "daily"}`

3. **Тестирование планировщика экспериментов**:
   - Endpoint: `POST /api/v1/ml/plan`
   - Body: 
     ```json
     {
       "experiment_description": "Анализ волатильности Solana",
       "available_data": {"symbols": ["SOL"], "timeframe": "1h", "period": "30d"},
       "constraints": {"execution_time": "fast"}
     }
     ```

### 4. Мониторинг производительности

Мониторинг работы ML-компонентов для проверки эффективности:

- **Логи**: Проверяйте `logs/wsl2_backend_*.log` для диагностики
- **Использование CPU/RAM**: Используйте `htop` в WSL2
- **Метрики моделей**: Проверяйте скорость обработки запросов в логах

### 5. Дальнейшие улучшения

1. **GPU-оптимизация**:
   - Если доступна NVIDIA GPU с поддержкой в WSL2, измените конфигурацию для её использования
   - Проверьте поддержку CUDA: `nvidia-smi` в WSL2

2. **Оптимизация моделей**:
   - Кэширование результатов частых запросов
   - Предварительная загрузка моделей при запуске
   - Асинхронная обработка пакетных запросов

3. **Интеграция с фронтендом**:
   - Визуализация ML-метрик и результатов
   - Интерактивные дашборды для экспериментов

## Советы по производительности

1. Если на WSL2 недоступна GPU, используйте:
   - Более легкие модели
   - Увеличьте таймауты для запросов
   - Мониторьте использование ОЗУ

2. Для предотвращения проблем с памятью:
   - Установите LOAD_IN_4BIT=True
   - Ограничьте количество одновременных запросов
   - Используйте очередь запросов при необходимости

## Заключение

WSL2 предоставляет существенные преимущества для ML-компонентов HTX проекта. С правильной настройкой и оптимизацией, вы можете использовать весь потенциал ML-анализа даже без специализированного оборудования.
